## A. Executive Summary

You have built a **working alpha** of a genuinely novel product. The architecture is sound, the code is real (6,186 lines across the memory layer alone), the tests are comprehensive (17 dedicated test files, 1006 passing), and the pipeline runs end-to-end. The three core bets — graph as structural truth, memory committed to Git, and task-aware context packs — are all implemented and wired together.

But running it live on this repo just now revealed the honest gap: **the output quality on real repos, including this one, is not yet "superpower" level.** The system works. The artifacts it produces are often wrong on substance, and `prepare-context` misfires on tasks that don't literally match a directory name. The graph — which is supposed to be the differentiator — is not built by default and most users won't have it when they first run `memory init`.

You are closer than you think to something genuinely good. But the gap between "it runs" and "it impresses" is real and specific.

## C. What the System Can Do Today — Honest Accounting

**Runs on this repo right now:**

```
repomind memory prepare-context "add rate limiting to the auth middleware"
→ Returns: code_review_graph/memory module + all 13 files + all 18 tests
→ Shows warning: "No specific area matched this task."
```

That's the honest output. On **this** repo with **this** task, the system returns noise — the entire memory module with no confidence, because "auth middleware" doesn't match any directory name in a developer tool repo.

**Where it works well:**

1. `memory init` on any repo: generates complete `.agent-memory/` tree with all artifacts. Runs in seconds. The pipeline is real and complete. ✓

2. `memory explain <module>`: Works correctly. `explain code_review_graph/memory` returns confidence score, all 13 files, all 18 tests, safe-boundary notes. The structure is right. ✓

3. `memory refresh`: The incremental planner works. After running `--full`, the freshness metadata is written, `changed` shows "Last refresh: 2026-03-25". ✓

4. `prepare-context` on a domain-structured repo (e.g., a web app with `src/auth/`, `src/billing/`, `src/api/` directories): Works well. The classifier detects the directories as features, the scorer matches "fix auth bug" → auth feature. This is the intended use case and it functions correctly.

5. Human overrides: `memory annotate` creates the scaffold, the YAML is loaded and applied. Works.

6. Auto-refresh: `repomind update` triggers memory refresh. Works.

7. Graph enrichment: When `graph.db` exists, `graph_bridge` adds import fan-in/out, test links, structural neighbors. The degradation when absent is perfect.

**Where it doesn't work well yet:**

1. **Flat single-package repos**: The classifier only detects subpackage boundaries (`__init__.py` in subdirectories). This repo's `code_review_graph/parser.py`, `graph.py`, `tools.py`, `incremental.py` are all flat files — the classifier sees one module, not five. Zero features detected on this repo.

2. **Responsibility inference**: `modules/code_review_graph-memory.md` says **"Responsibilities: data models / ORM definitions"** for the memory engine module. This is wrong and would mislead Claude.

3. **Architecture artifacts**: `architecture.md` says `benchmarks/` is "production source code." True in a directory-walking sense, wrong in meaning. The two-layer architecture (graph engine vs memory subsystem) is invisible to it.

4. **Conventions bleed**: `rules/conventions.md` lists Go, Java, Rust, TypeScript conventions for what is overwhelmingly a Python project — because the test fixtures for those languages exist in `tests/fixtures/`.

5. **`changed` with no prior refresh**: Returns "No refresh data found. Run `memory refresh`." First-time UX is broken.

6. **prepare-context keyword misfires**: The scorer is purely lexical. "add rate limiting" matches `code_review_graph/memory` only because nothing else matched at all — the fallback to broadest context is unhelpful when the task is completely unrelated to the only module the classifier found.

## F. How Much of the Fork's Power We're Actually Using

**Using well:**
- `GraphStore.get_impact_radius()` — used in graph_bridge for BFS expansion in `changed`, `prepare-context`, and refresh planning. This is the fork's core capability and we use it correctly.
- `GraphStore.get_edges_by_target()` — TESTED_BY edges for test file discovery. Real use.
- `GraphStore.get_edges_by_source()` — IMPORTS_FROM for structural neighbors. Real use.
- `GraphStore.get_stats()` — availability check. Correct.
- `GraphStore.get_nodes_by_file()` — file → node enumeration for signal gathering. Real use.

**Underusing significantly:**

1. **Node content is completely ignored.** The graph stores function names, class names, docstrings (where parseable), and call sites for every file. The memory layer reads only edge relationships — it never reads what functions are named `authenticate()`, `validate_token()`, `rate_limit_middleware()`. This means the generator produces "Responsibilities: data models / ORM definitions" by guessing from file stem names (`models.py` → "data models") instead of reading what the file actually contains.

2. **`query_graph_tool` with `file_summary` pattern** — this gives real structural summaries of files. The memory layer never calls it. The generator could use this to produce accurate responsibility descriptions instead of stem heuristics.

3. **`semantic_search_nodes_tool`** — vector-based entity search. Not used by `prepare-context` at all. The scorer is 100% keyword/token matching. If semantic search were wired into `prepare-context`, "fix the token expiry edge case" would surface `authenticate()`, `validate_token()` → auth module, even without keyword overlap.

4. **Function/class name vocabulary** — the graph contains every function and class name in the repo. None of this is used to improve classification confidence, responsibility inference, or context matching.

5. **`get_review_context_tool`** — not used by the memory layer. The review tool assembles structural source context with impact; memory could reuse this for `explain` enrichment.

**Biggest missed opportunity:** The graph has AST-level node content — function names, class names — for every file. If the classifier and generator read this vocabulary instead of just using edge connectivity and file stems, responsibility inference would be accurate, features would be detected even in flat packages, and `prepare-context` matching would be semantic rather than lexical.

## G. Biggest Gaps

**Technical gaps:**

1. **No node-content reading in classifier/generator.** The classifier uses `get_impact_radius()` for edge signals but never reads node names. The generator infers "data models" from `models.py` file stem. Both should be reading function/class names from the graph nodes to produce real descriptions.

2. **Flat-package classification blind spot.** `classifier.py` detects features and modules only when there are directory-level boundaries (subdirectories with `__init__.py`, `index.ts`, etc.) or top-level domain keywords in directory names. A project with `src/auth.py`, `src/billing.py`, `src/api.py` as flat files gets zero features detected. This is very common.

3. **`prepare-context` fallback is noise.** When no feature/module scores above `_MIN_SCORE = 0.05`, the code falls back to returning the full module list with a warning. This should instead use graph similarity or return nothing with a better message.

4. **`changed` requires prior refresh.** "No refresh data found" is a blocking message for a first-time user. It should either run a lightweight refresh inline or produce useful heuristic output without prior metadata.

5. **Conventions bleed from test fixtures.** Language detection includes test fixture files in other languages, producing Go/Java/Rust conventions in a Python project. The conventions generator should filter by lines-of-code weighting.

**Product gaps:**

1. **The setup order is fragile.** `memory init` before `repomind build` gives heuristic-only output and the user doesn't know the difference. The gap between "heuristic mode" and "graph-enriched mode" is enormous and invisible.

2. **Artifact quality on real repos is inconsistent.** A domain-structured web app gets good output. A flat single-package tool gets "data models / ORM definitions" for its core engine. There's no quality bar enforcement.

3. **No iteration loop.** After the user runs `memory init` and sees wrong artifacts, there's no guided path to fix them. `memory annotate` exists but most users won't find it or know what to put in it.

4. **`prepare-context` doesn't say what it matched or why.** The output shows files but doesn't explain which feature matched and at what score. The user has no way to know if the result is "high confidence" or "I couldn't find anything, here's everything."

**Trust/UX gaps:**

1. **Responsibility descriptions are visibly wrong.** If a developer sees "Responsibilities: data models / ORM definitions" for the memory engine, they immediately lose trust in everything else the system generated.

2. **First cold run is weak.** `memory init` on a flat repo produces: 0 features, 1 module, wrong responsibility, generic conventions, thin architecture doc. That's not impressive enough to make someone commit `.agent-memory/` to Git.

3. **The graph-enriched experience (the good one) is opt-in and undiscoverable.** The README says "graph-assisted context packs" but the experience most users get first is pure heuristics.

## H. Highest-Leverage Next Steps

**3 areas for biggest jump in product quality:**

1. **Read graph node names in the generator and classifier.** Add a `get_file_vocabulary(files, repo_root) -> dict[str, list[str]]` function to `graph_bridge.py` that returns function and class names per file. Feed this into `generator.py`'s responsibility inference instead of file-stem heuristics. Also feed into `context_builder.py` scoring so "fix token expiry" matches `authenticate()`, `validate_token()`. This single change would make the output look real, not generated.

2. **Fix flat-package feature/module detection.** The classifier needs a fallback for repos where all source is in a single flat package. Strategy: use the graph's IMPORTS_FROM edges to cluster highly-connected flat files into logical groups, treating clusters as implied modules. Without the graph, fall back to file-stem domain-keyword matching (e.g., `auth.py`, `billing.py`, `payments.py` → detected as features). This fixes the classification gap that makes this tool's own artifacts wrong.

3. **Improve the first cold run so it impresses.** Three concrete changes: (a) detect whether `graph.db` exists before init and print "For best results, run `repomind build` first" prominently at the top, not buried; (b) enforce a minimum-quality check — if 0 features detected, say so clearly with actionable guidance; (c) the module description should show file count + top function/class names from the graph, not just a stem heuristic.

**3 areas for biggest jump in launch readiness:**

1. **Fix the `changed` first-run experience.** Right now it says "No refresh data found" which is a dead end. It should detect the freshness gap and either run a lightweight refresh automatically or produce a heuristic-only output with a clear "run `memory refresh` for richer data" note. The current message is a broken first impression.

2. **Make the setup order enforced and guided.** The `memory init` command should check if `graph.db` exists. If it doesn't, print a prominent step-one notice at the top: "Step 1 (recommended): `repomind build` — enables graph-assisted classification and impact analysis." After that, "Step 2: `repomind memory init`." Two commands, guided.

3. **Add a quality signal to the output.** When `memory init` finishes, show "0 features detected" alongside "1 module detected" so the user knows something is sparse and what to do about it. Right now the output shows counts but not whether they're good or bad. Add a simple quality verdict: "Good: 3 features + 5 modules" vs "Sparse: 0 features — run `memory annotate` to add domain corrections."

**Single next step:**

Wire graph node names (function names, class names per file) into the generator's responsibility inference and the context builder's relevance scoring. This is the single change that most directly closes the gap between "system runs" and "output is trustworthy." It uses the fork's actual AST data that is already sitting in `graph.db` unused, it fixes the worst visible output quality problem (wrong responsibility labels), and it makes `prepare-context` matching semantic rather than lexical. Everything else is UX polish by comparison.
